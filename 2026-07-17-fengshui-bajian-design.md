# 风水摆件工具 设计文档

- 日期：2026-07-17
- 状态：待评审
- 所属专题：算命占卜 / 风水布局
- 关联工具：玄空风水（角度罗盘引擎）、紫白飞星、BaZi 计算引擎、办公室风水布局（office-fengshui）

## 1. 背景与目标

在「风水布局」专题下新增一个动作导向型工具：给定一个房间，告诉用户具体该在哪个角落摆什么风水摆件、为什么、以及不摆全套的话优先摆哪一个。

跟已有的玄空风水、紫白飞星、办公室风水布局工具的区别：那几个工具的产出是「诊断」（这个方位是什么星、吉凶如何），这个工具的产出是「动作」（诊断之后，具体放什么东西）。定位上它是消费上述工具/引擎输出的下游功能，而不是重新造一遍轮子。

## 2. 已锁定的关键决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| 输入精度 | 精装版：复用玄空风水工具的角度罗盘输入（精确朝向角度 + 建成/装修年份） | 只有起出完整宅飞星盘，财位/文昌位这类词才是真按这套房子精算出来的，专业感是这个工具的核心卖点 |
| 使用者数量 | 统一支持多人：任意房间类型都可增删使用者，各自独立标注，不做取舍合并 | 个性化图层（八宅游星、命理专属位）依赖命卦和 BaZi 日主，同一房间不同人结果天然不同，强行合并没有意义 |
| 摆件产出方式 | 混合：规则库判定五行缺口/煞气类型，AI 在判定框架内挑选具体物件并转译成人话 | 规则库保证同一方位+同一星曜组合的判定结果稳定可复现；AI 负责的是"选哪个具体物件、怎么讲"这层，兼顾一致性和文案新鲜感 |
| 综合评分 | 不做加权总分，分层展示（环境层 + 逐人层并列） | 多人场景下同一方位对不同人可能吉凶相反，强行出一个总分是伪精确 |
| 起盘范围 | 以房间自身为独立起盘单位，用房间自己的朝向角度和中心点起盘，不建模整栋房屋 | 玄空宅飞星传统上覆盖整栋房子，但本工具及既有的 office-fengshui 工具一直按房间类型独立处理，房间自成单位能避免要求用户填写整栋房屋朝向这类拿不到的数据，也让"房间内部精细切分"这个需求域收敛到房间自身 |
| 房间跨宫位处理 | v1 即做精细分区：矩形长宽 + 门的大致方位为必填输入，用45度角辐射法从房间中心点切八个方位扇区；异形户型追加"缺角/凸出标注"作为可选进阶输入 | 矩形+角度输入比自由画布描边工程成本低得多，且与整套工具"精确罗盘角度"的定位保持一致；缺角标注复用玄空实操里本来就有的判定规则，不需要另造多边形编辑器 |
| 跨工具数据复用 | 不允许，每次独立填写，不从其他工具的本地存储自动带入朝向/生辰 | 明确不做，保持输入链路独立简单 |

## 3. 输入设计

复用玄空风水工具的输入字段，保持体验一致：

- 房间类型（沿用 office-fengshui 的房间类型选择器：卧室 / 书房 / 办公室 / 客厅 / 店铺 等，做成通用组件）
- 房间精确朝向角度（沿用玄空风水工具的罗盘输入组件；本工具以房间自身为起盘单位，见第 4.1 节）
- 建成/装修年份（用于起宅飞星、定元运）
- 房间长、宽（矩形近似，用于定中心点和八方位扇区边界）
- 门的大致方位（八方位选一，必填）
- 异形户型可选标注：从矩形四角/四边勾选"缺角"或"凸出"
- 使用者列表（可增删，每人填：昵称、出生年月日时、性别；昵称仅用于前端标注，不采集真实姓名）
- 当前年月：系统自动取，无需用户输入

无登录系统，沿用浏览器本地持久化存储的既有惯例；同一房间的输入可以保存复用，避免每次重新填。不与其他工具的本地存储互通，每次独立填写。

## 4. 命理计算逻辑（calc.post.ts）

分两层，环境层不依赖使用者，逐人层依赖每个使用者的生辰。

### 4.1 环境层

**房间几何与九宫切分**：以房间自身为独立起盘单位（不建模整栋房屋）。用房间长宽定几何中心点，用45度角辐射法（而非三等分法，与全工具"精确罗盘角度"的定位保持一致）从中心点切出N/NE/E/SE/S/SW/W/NW八个方位扇区。中心点本身对应太极位，实操中不安放催旺类摆件，只做留白或化解处理。若用户标注了缺角/凸出，对应方位额外叠加"气场受损"判定，走单独的化解规则。

**玄空宅飞星**：房间朝向角度 + 建成年份 → 定元运 → 起山星向星，得出上述八个方位扇区各自的山星、向星数值和当运旺衰判定。

**紫白流年飞星**：当前年月 → 起流年九宫图，叠加到宅飞星盘上。重点标记太岁方、五黄方、三煞方——这几个是当年需要避让而非催旺的煞位，需要单独打标，防止被误用成招财位（这是行业常见的产品坑）。

**五行缺口判定（规则库）**：基于上述叠加结果，对每个方位扇区判定其五行调理方向，例如二黑/五黄病符坐镇 → 判定"需要金泄土"；旺星失位或被克 → 判定"需要对应五行催旺"；太岁/三煞方或缺角方位 → 判定"只化不催"。这一层只输出判定类型（如 `metal_drain` / `water_boost` / `avoid_only`），不指定具体物件，具体物件留给 AI 层挑选。

### 4.2 逐人层（数组，每个使用者一条记录）

**八宅游星**：出生年份 + 性别 → 命卦 → 结合房间坐向，定出该使用者的四吉（生气/天医/延年/伏位）四凶（五鬼/六煞/祸害/绝命）八方位分布，判断当前房间落在其游星体系的哪个位。

**命理专属位**：日主/年支 → 查表得出文昌贵人位、桃花位（三合局）、天乙贵人位，与房屋朝向无关，是"人的方位"，标注房间是否命中。

## 5. calc.post.ts 输出结构（草案）

```typescript
interface FengshuiOrnamentCalcResult {
  roomGeometry: {
    doorDirection: string // 如 "东南"
    irregularCorners?: Array<{ direction: string; type: 'missing' | 'protruding' }>
  }
  environment: {
    // 八个方位扇区，以房间自身中心点 + 45度角辐射法切分，非整栋房屋盘
    palaces: Array<{
      direction: string // 如 "东南"
      mountainStar: number
      facingStar: number
      isTaiSui: boolean
      isWuHuang: boolean
      isSanSha: boolean
      hasIrregularCorner: boolean
      elementGap: 'metal_drain' | 'water_boost' | 'wood_boost' | 'fire_boost' | 'earth_boost' | 'avoid_only' | 'neutral'
    }>
  }
  perPerson: Array<{
    nickname: string
    baguaAssignment: Record<string, 'shengqi' | 'tianyi' | 'yannian' | 'fuwei' | 'wugui' | 'liusha' | 'huohai' | 'jueming'>
    matchedPositions: {
      wenchang: boolean
      taohua: boolean
      guiren: boolean
    }
  }>
}
```

不含综合评分字段，环境层和逐人层并列输出，前端和 AI 层各自消费。

## 6. AI reading.post.ts 设计

沿用统一的"幽默隐士"人设，输入是 calc 已经判定好的结构化数据，AI 只做两件事：在规则库给定的五行框架内挑选具体摆件（例如同样判定 `metal_drain`，这次给铜铃铛下次给白水晶都可以，避免机械重复）；把冷数据转译成人话建议。

输出分层：

1. 一句话整体定性开场（不上来就甩数据）
2. 逐方位摆件建议：放哪个角落、放什么、为什么——具体到可执行动作
3. 煞位避坑提醒：太岁/五黄/三煞方专门用调侃语气提醒"只化不催"，防止用户好心办坏事
4. 多人冲突时如实说明，不擅自替用户拍板（如"这个位置对A是延年对B是六煞，你俩自己商量"）
5. 优先级建议：没空全屋摆的话，先补哪一个方位——这是工具区别于纯资料查询站的核心实用价值

专业依据（宅飞星数值、游星归类等）默认折叠，保留一个可展开的"为什么"区，兼顾专业感和轻量体验。

## 7. 与现有工具的关系

- 复用玄空风水工具的罗盘角度输入组件和起盘引擎，不重新实现
- 复用紫白飞星工具的流年起盘逻辑
- 复用 BaZi 引擎计算命卦、日主、年支
- 遵循现有 dual-endpoint SSE 架构（calc.post.ts 出确定性结构化数据，reading.post.ts 走 AI 解读）
- 不引入新的付费 API，AI 部分走现有的服务端调用惯例
- 房间类型选择器沿用/扩展 office-fengshui 的既有组件，避免重复造轮子
